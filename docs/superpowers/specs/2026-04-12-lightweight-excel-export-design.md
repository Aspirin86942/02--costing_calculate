# 轻量 Excel 导出性能优化设计

## 1. 背景

当前成本核算 ETL 的单次全量运行已经可以正确完成，但导出阶段耗时过高，导致整体处理速度偏慢。

基于 2026-04-12 在本地对 `gb` 和 `sk` 原始样本的实际运行观察：

- 原始 Excel 读取耗时约 38 秒；
- 清洗、拆表、聚合、异常分析等计算阶段耗时约 8 到 10 秒；
- Excel 写出阶段耗时接近 3 分 50 秒，是当前主要瓶颈。

这说明当前优先问题不是“计算逻辑不够快”，而是“Excel 渲染过重”。

## 2. 目标

本次优化目标是：在仍然导出 `.xlsx` 的前提下，显著降低导出阶段耗时，并保持业务上必须可用的最小展示能力。

本次明确保留的输出能力：

- 固定 sheet 清单不变；
- 必要表头保留；
- 数字格式保留；
- 自动筛选保留；
- 冻结窗格保留；
- `按工单按产品异常值分析` sheet 的异常颜色高亮保留。

本次明确允许牺牲的内容：

- 大部分单元格边框；
- 大部分填充色；
- 大部分逐单元格对齐控制；
- 复杂的列宽估算；
- 除 `按工单按产品异常值分析` 外的颜色提示；
- 为视觉排版服务的额外逐格写入逻辑。

## 3. 非目标

本次不作为目标处理的事项：

- 不先重构 ETL 业务计算逻辑；
- 不先改异常判定口径；
- 不减少现有 sheet 数量；
- 不改为非 Excel 最终交付格式；
- 不为了提速而改变关键数据值、行数、错误日志口径。

## 4. 方案选择

评估过的主要方案：

### 4.1 保留 `openpyxl`，只削减样式

优点是改动小、风险低；缺点是仍受 `openpyxl` 大量写单元格性能限制，提速空间有限。

### 4.2 改为轻量 `xlsxwriter` 导出

优点是更适合“少样式、高吞吐”的 `.xlsx` 输出场景，并且能用条件格式替代部分逐格着色逻辑；缺点是需要重做导出层的部分实现。

### 4.3 计算与导出解耦

这有利于后续开发效率，但对“单次全量导出”本身不是第一收益点。

### 4.4 结论

本次采用 **方案 4.2：轻量 `xlsxwriter` 导出**，并默认开启 `constant_memory` 模式，以降低大 sheet 写出时的内存占用并提高稳定性。分析与 ETL 的输入输出契约保持不变。后续如需继续提升迭代效率，再单独设计“计算与导出解耦”。

## 5. 设计原则

### 5.1 正确性优先

性能优化不得改变以下业务结果：

- 各 sheet 的行数；
- 关键金额列；
- `error_log` 的记录数与异常分类口径；
- `按工单按产品异常值分析` 中的异常标记结果。

### 5.2 导出层只负责渲染

导出层只消费分析阶段已经准备好的 DataFrame / FlatSheet / SectionBlock 数据，不在写出阶段再做业务计算，避免把“性能逻辑”和“业务逻辑”混在一起。

### 5.3 样式必须服务于识别，不服务于装饰

保留的样式只为三件事服务：

- 用户看得懂列名；
- 用户看得懂数字；
- 用户能快速看到工单异常高亮。

除此以外不再为“看起来更像报表”增加大规模单元格处理。

## 6. 架构设计

### 6.1 保持现有上游契约

以下模块的输入输出契约保持不变：

- `src/etl/costing_etl.py`
- `src/etl/pipeline.py`
- `src/analytics/qty_enricher.py`
- `src/analytics/table_rendering.py`
- `src/analytics/anomaly.py`

也就是说，上游仍然产出：

- `detail_df`
- `qty_sheet_df`
- `analysis_tables`
- `work_order_sheet`
- `product_anomaly_sections`
- `error_log`

### 6.2 重构导出层

导出层改为“轻量渲染器”，重点替换以下区域：

- `src/excel/workbook_writer.py`
- `src/excel/sheet_writers.py`

重构后的导出职责：

- 普通 DataFrame sheet：快速整表写出；
- 分析 sheet：保留扁平结构，不做复杂装饰；
- 工单异常 sheet：保留必要格式和异常高亮；
- `error_log`：直接输出，不追加重样式。

### 6.3 写出路径分层

导出层需要支持两级写出路径：

#### A. 默认路径：保留 Pandas 入口

对中小体量 sheet，仍允许使用 `pandas.ExcelWriter(..., engine='xlsxwriter')` 写出，以降低迁移复杂度。

要求：

- `engine_kwargs` 中默认启用 `constant_memory`；
- 写出后只做列级格式、筛选、冻结窗格和必要条件格式；
- 不再恢复逐单元格样式循环。

#### B. 热点路径：直接使用原生 `xlsxwriter`

对已证明是热点的大 sheet，允许绕过 `DataFrame.to_excel()`，直接使用 `worksheet.write_row()` 流式写出。

优先候选 sheet：

- `成本明细`
- `产品数量统计`
- `error_log`

实现要求：

- 行数据优先通过 `DataFrame.itertuples(index=False, name=None)` 迭代；
- 不使用 `DataFrame.to_dict('records')`，避免生成大量 Python `dict` 带来的额外内存与对象分配开销；
- 先写表头，再按行顺序写数据，满足 `constant_memory` 的顺序写约束。

### 6.4 样式分级

导出层按两类 sheet 区分策略：

#### A. 普通轻量 sheet

适用范围：

- `成本明细`
- `产品数量统计`
- `直接材料_价量比`
- `直接人工_价量比`
- `制造费用_价量比`
- `按产品异常值分析`
- `error_log`

策略：

- 仅保留表头样式；
- 保留数字列格式；
- 保留筛选；
- 保留冻结窗格；
- 列宽使用固定宽度或仅按表头粗略估算；
- 不再逐单元格应用边框、填充、对齐。

#### B. 工单异常重点 sheet

适用范围：

- `按工单按产品异常值分析`

策略：

- 保留表头；
- 保留数字格式；
- 保留筛选；
- 保留冻结窗格；
- 保留异常颜色高亮；
- 仍避免除高亮外的逐单元格重样式。

## 7. 异常高亮设计

### 7.1 高亮保留范围

只保留 `按工单按产品异常值分析` 中的异常颜色提示。

### 7.2 实现方式

不再沿用 Python 逐行逐格设置颜色的方式，改为由 Excel 条件格式完成高亮。

条件格式基于“值列 + 异常标记列”成对配置，至少覆盖当前已有的高亮关系：

- `直接材料单位完工成本` / `直接材料异常标记`
- `直接人工单位完工成本` / `直接人工异常标记`
- `制造费用单位完工成本` / `制造费用异常标记`
- `制造费用_其他单位完工成本` / `制造费用_其他异常标记`
- `制造费用_人工单位完工成本` / `制造费用_人工异常标记`
- `制造费用_机物料及低耗单位完工成本` / `制造费用_机物料及低耗异常标记`
- `制造费用_折旧单位完工成本` / `制造费用_折旧异常标记`
- `制造费用_水电费单位完工成本` / `制造费用_水电费异常标记`

高亮规则：

- 标记为 `关注` 时，使用黄色背景；
- 标记为 `高度可疑` 时，使用红色背景，必要时白字；
- 其他值不着色。

条件格式实现约束：

- 条件格式在目标 sheet 数据写完后、workbook 关闭前统一挂载；
- 公式引用必须使用正确的绝对列引用，避免整段区域只有首行判断正确、其余行错位；
- 高亮规则按“值列区域”和“标记列公式”成对配置，不再对单元格逐个着色。

### 7.3 为什么这样做

这样做的原因不是“代码更好看”，而是：

- 条件格式由 Excel 引擎执行，通常比 Python 逐格刷样式更省时；
- 高亮逻辑集中在规则定义上，维护成本更低；
- 用户打开文件后仍能直接看到异常重点。

### 7.4 格式对象复用

所有导出格式对象必须在 workbook 初始化阶段集中创建并复用，例如：

- 表头格式；
- 数字格式；
- `关注` 高亮格式；
- `高度可疑` 高亮格式。

禁止在行循环或单元格循环内反复创建格式对象。

## 8. 数据流

优化后的数据流如下：

1. 读取原始 Excel；
2. 执行清洗、拆表、聚合、异常分析；
3. 生成导出所需的数据对象；
4. 轻量导出器一次性写出各 sheet；
5. 对工单异常页挂载条件格式；
6. 保存最终 `.xlsx` 文件。

关键变化在第 4 和第 5 步：导出阶段从“大量单元格操作”改为“整表写出 + 少量列格式 + 条件格式”。

## 9. 兼容性与风险

### 9.1 预期变化

用户会看到以下变化：

- 文件视觉效果明显更简洁；
- 某些旧版样式不再出现；
- 列宽可能不再像当前版本那样贴合内容；
- 除工单异常页外，其他 sheet 不再承担视觉提示职责。

### 9.2 主要风险

主要风险不是业务计算错误，而是导出表现差异：

- 某些表格排版与旧文件不一致；
- 某些数字列可能因格式配置遗漏而显示为普通文本；
- 条件格式应用范围如果配置错误，可能导致高亮缺失或误着色。
- 精确耗时改善幅度受磁盘写入、ZIP 打包和 Python 对象转换开销影响，不能在实现前承诺固定秒数。

### 9.3 风险控制

必须通过自动化或半自动核验控制以下事项：

- 各 sheet 名称与顺序正确；
- 各 sheet 行数与旧实现一致；
- 关键数值列总和一致；
- `error_log` 行数一致；
- 工单异常标记列值一致；
- 条件格式规则已成功写入目标区域。

## 10. 观测与验收

### 10.1 观测要求

实现时需要记录至少三段耗时：

- 读取耗时；
- 计算耗时；
- 导出耗时。

这样做的原因是：后续如果性能仍不达标，必须知道慢在哪里，而不是重新回到凭感觉优化。

### 10.2 验收标准

本次优化完成后，以以下标准验收：

- `pytest` 全量通过；
- 真实 `gb`、`sk` 样本都能成功导出；
- 导出的 8 张 sheet 都存在；
- 关键业务数据与旧版本一致；
- `按工单按产品异常值分析` 的异常高亮存在；
- 总耗时明显下降，且导出阶段耗时明显低于当前实现。

“明显下降”必须以同一批输入文件的前后对比结果说明，不能只凭体感判断。
本次不把“20 秒以内”作为承诺性目标，避免在缺少实测证据的情况下过度承诺。

## 11. 实施边界

本次实现允许修改的重点文件：

- `src/excel/workbook_writer.py`
- `src/excel/sheet_writers.py`
- 可能补充一个新的轻量导出辅助模块，例如 `src/excel/fast_writer.py`
- 必要时微调 `src/excel/styles.py` 以保留最少样式常量
- 必要时在导出流程中补充热点 sheet 的原生 `xlsxwriter` 写出辅助逻辑

本次实现原则上不应主动改动：

- `src/etl/` 下业务清洗与拆表逻辑；
- `src/analytics/` 下异常判定与金额聚合口径；
- `tests/contracts/` 中已确认的业务契约，除非为了反映“样式降级但数据不变”的新预期而做最小必要更新。

## 12. 后续计划建议

若本次导出层优化完成后性能仍不满足预期，再考虑第二阶段工作：

- 为开发调试增加“计算与导出解耦”；
- 对分析阶段热点做 profiling；
- 仅对已证明是瓶颈的 `iterrows` / `apply` / `Decimal` 聚合路径做局部重写。

顺序不能反过来。因为在当前证据下，先重构计算核心的投入产出比不高。

## 13. 额外内存控制建议

本次不把“全流程内存治理”作为主要目标，但实现导出时应遵守以下原则：

- 一个大 sheet 写完后，如果对应 DataFrame 后续不再使用，可以及时释放引用；
- 避免把同一份大表同时保留为 DataFrame、`dict` 列表和写出缓存三份结构；
- 如需补充内存观测，应优先在导出阶段记录对象规模和阶段性耗时，而不是先引入复杂的内存优化框架。
