# 大表导出与日志收敛优化设计

## 1. 背景

当前项目已经完成一轮轻量 `xlsxwriter` 导出改造，整体链路可以稳定跑通，但在大样本下，导出阶段仍然是主要瓶颈。

基于 **2026-04-13** 对用户刚替换后的真实样本实测：

### `gb`

- 输入文件：`gb-成本计算单_2026041311462296_100160.xlsx`
- `成本明细输入行数=58,007`
- `产品数量统计输出行数=885`
- `export=7.278s`
- `total=9.112s`

### `sk`

- 输入文件：`sk-成本计算单_2026041311461807_3592191.xlsx`
- `成本明细输入行数=425,459`
- `产品数量统计输出行数=11,239`
- `export=56.778s`
- `total=76.202s`

这说明两件事：

1. 当前主瓶颈已经非常明确，是 **超大明细表写出 `.xlsx`**，而不是分析计算阶段。
2. `gb` 当前样本已经处于可接受区间，真正需要优化的是 **大体量 `sk`** 的导出路径。

用户已明确新的约束：

- `gb/sk` 两条管线都 **不再生成** `*_处理后.log`
- 控制台质量摘要 **必须保留**
- 仍然需要输出 **一个完整的大 Excel**
- 允许做 **大表导出路径优化**
- 允许做 **大表样式轻量化**
- 不接受把超大明细改为单独 `CSV/Parquet` 代替 Excel 主产物

## 2. 目标

本次优化目标如下：

### 2.1 功能目标

- 停止生成 `*_处理后.log`
- 保留控制台质量摘要输出
- 保留 `*_处理后.xlsx`
- 保留 `*_处理后_error_log.csv`

### 2.2 性能目标

- 以相同真实样本为基线，显著降低 `sk` 的 `export` 阶段耗时
- 优化重点放在 `成本明细`、`产品数量统计`、`error_log` 三类平铺大表
- 不把优化重点放在分析计算逻辑重写上

### 2.3 兼容目标

以下外部契约保持稳定：

- workbook 的 sheet 名称与顺序不变
- 各 sheet 的关键列、数据口径、行数口径不变
- 冻结窗格保留
- 自动筛选保留
- 数值格式保留
- `按工单按产品异常值分析` 的异常高亮保留
- `error_log` 仍保留在 workbook 中，同时继续导出 CSV

## 3. 非目标

本次不做以下事项：

- 不重写全链路 ETL 业务逻辑
- 不改变白名单、勾稽、异常判定或独立成本项口径
- 不减少现有 8 张 sheet
- 不把 `成本明细` 从 Excel 主产物中移除
- 不引入新的最终交付格式替代 `.xlsx`
- 不为了性能而牺牲金额精度或数据正确性

## 4. 方案选择

本次讨论过 3 种方案：

### 4.1 方案 A：仅停掉 `.log` 文件

优点：

- 改动很小
- 风险很低

缺点：

- 对总体时间改善非常有限
- 无法解决 `sk export=56.778s` 的主瓶颈

### 4.2 方案 B：停掉 `.log` 文件 + 热点大表快写 + 大表样式轻量化

优点：

- 收益集中命中真正瓶颈
- 不改业务口径
- 仍保留一个完整 Excel
- 风险可控，改动边界主要集中在导出层

缺点：

- workbook 的大表视觉会比现在更朴素
- 需要补充导出层路由和样式分级

### 4.3 方案 C：把超大明细移出 Excel，只保留汇总分析页

优点：

- 性能收益最大

缺点：

- 与用户约束冲突
- 改变交付方式

### 4.4 结论

本次采用 **方案 B**：

- 停掉 `*_处理后.log`
- 保留控制台摘要
- 保留完整 Excel
- 对热点大表使用更快的写出路径
- 对热点大表采用轻量样式

## 5. 设计原则

### 5.1 正确性优先于视觉效果

导出优化不能改变以下内容：

- 行数
- 金额
- 单位成本
- 勾稽字段结果
- `error_log` 记录数和异常类型
- 异常高亮对应的异常标记结果

### 5.2 只优化热点，不扩散改动

只有已证明耗时高的大表才使用专用快写路径。分析页、异常页和兼容摘要页尽量沿用当前契约，避免整套导出层再次大范围重做。

### 5.3 样式必须服务可用性，不服务装饰

大表保留的样式只服务于：

- 看懂表头
- 看懂数字
- 维持冻结窗格和筛选

不再为大表数据区逐格增加边框、填充和复杂对齐。

### 5.4 控制台日志保留可观测性

停掉 `*.log` 文件不等于停掉质量观测。控制台摘要仍然是必需品，用于：

- 快速确认输入输出路径
- 查看行数勾稽
- 查看空值率与重复键
- 查看 `error_log_count`

## 6. 架构设计

本次只调整导出层和外层产物控制，不动上游业务口径。

### 6.1 外层产物调整

当前 `runner.py` 会同时：

- 打印质量摘要到控制台
- 写入 `*_处理后.log`
- 写出 `*_处理后_error_log.csv`

改造后：

- 继续打印控制台质量摘要
- 停止写入 `*_处理后.log`
- 继续写出 `*_处理后_error_log.csv`

也就是说，质量摘要文本仍会被构造，但只用于 `print()`，不再落盘。

### 6.2 导出层分成两类路径

#### A. 热点大表快写路径

适用 sheet：

- `成本明细`
- `产品数量统计`
- `error_log`

特点：

- 直接消费完整 frame
- 使用 `write_dataframe_fast()` 这类按行流式写出的热点路径
- 尽量避免通用 `SheetModel -> rows_factory -> 二次覆盖写数值列` 的高开销路径

#### B. 通用展示路径

适用 sheet：

- `直接材料_价量比`
- `直接人工_价量比`
- `制造费用_价量比`
- `按工单按产品异常值分析`
- `按产品异常值分析`

特点：

- 继续使用当前通用 `SheetModel` 机制
- 保留必要格式与条件格式
- 不对这些 sheet 追求极限吞吐

### 6.3 关键设计点：热点 sheet 必须保留对原始 frame 的直接访问

当前 `SheetModel` 主要暴露的是：

- `columns`
- `rows_factory`
- `number_formats`
- `freeze_panes`

这足以支持通用导出，但不利于热点 sheet 直接走更快的 DataFrame / Polars 写出路径。

因此本次需要给导出契约补充一个“热点写出提示”，设计上可接受两种落地方式：

#### 方式 1：扩展 `SheetModel`

增加字段，例如：

- `write_mode`
- `source_frame`
- `style_profile`

优点：

- 不需要引入第二套 payload
- `write_workbook_from_models()` 仍然是统一入口

缺点：

- `SheetModel` 会同时承担“展示契约”和“导出策略”两类信息

#### 方式 2：新增导出计划对象

例如新增 `WorkbookExportPlan` / `TabularSheetPlan`：

- 平铺热点表用 frame 直写计划
- 复杂展示表继续用 `SheetModel`

优点：

- 职责更干净

缺点：

- 需要引入额外契约对象和路由层

#### 本次推荐

本次推荐 **方式 1：扩展 `SheetModel`**。

原因：

- 这是一次导出层局部优化，不是再次全量重构
- 统一入口更利于保持当前链路稳定
- 可以在最小改动下让热点表拿到直写能力

## 7. 热点大表优化设计

### 7.1 现状问题

当前通用路径存在两个典型成本：

1. 热点大表被先按整行写一次，再对数值列逐格覆盖格式
2. 大表数据区仍承担了过多视觉格式职责

对几十万行数据，这两个点都会显著放大 Python 层调用次数。

### 7.2 优化策略

热点表统一采用以下策略：

- 表头单独写
- 数据区按行顺序流式写出
- 数字格式优先通过列级默认格式处理
- 尽量避免先写文本再逐格覆写数值
- 保持 `constant_memory=True`
- 保持 `strings_to_urls=False`

### 7.3 预期收益来源

不是依赖某一个小技巧，而是来自三个叠加：

1. 减少逐单元格 Python 调用
2. 减少同一数值单元格的重复写入
3. 减少大表数据区样式对象应用次数

## 8. 大表样式轻量化设计

### 8.1 适用范围

仅对以下大表做轻量化：

- `成本明细`
- `产品数量统计`
- `error_log`

### 8.2 明确保留

- 表头样式
- 数字格式
- 冻结窗格
- 自动筛选
- 固定宽度或低成本列宽策略

### 8.3 明确弱化

- 大表数据区逐格边框
- 大表数据区逐格对齐
- 大表数据区逐格填充
- 复杂列宽估算

### 8.4 不轻量化的部分

以下内容尽量保持现状：

- `按工单按产品异常值分析` 的异常高亮
- 分析页的分段结构
- `按产品异常值分析` 的兼容摘要页结构

## 9. 数据流

改造后数据流保持简单：

1. `pipeline.build_workbook_payload()` 继续生成 workbook payload
2. payload 中的热点 sheet 带上原始 frame 和写出模式
3. `workbook_writer.write_workbook_from_models()` 根据 `write_mode` 路由：
   - 热点大表 -> 快写路径
   - 其余 sheet -> 通用路径
4. workbook 写出完成后，`runner.py` 打印质量摘要并写出 `error_log.csv`
5. 不再生成 `*_处理后.log`

## 10. 文件级变更边界

本次实现的主要改动范围应控制在以下文件：

- `src/etl/runner.py`
- `src/analytics/contracts.py`
- `src/analytics/presentation_builder.py`
- `src/excel/workbook_writer.py`
- `src/excel/fast_writer.py`

必要时允许少量调整：

- `src/excel/styles.py`
- `tests/contracts/` 中与样式轻量化直接相关的最小必要断言

原则上不主动修改：

- `src/etl/` 中业务清洗与拆表逻辑
- `src/analytics/` 中金额、勾稽、异常口径

## 11. 风险与控制

### 11.1 风险：大表样式变化触发契约测试失败

控制方式：

- 提前明确哪些样式契约必须保留，哪些允许弱化
- 只更新与“大表数据区轻量化”直接相关的最小必要断言

### 11.2 风险：热点路径和通用路径的数据口径不一致

控制方式：

- 热点路径只改变写出方式，不改变输入 frame
- 对比 workbook 关键 sheet 的行数、列名和关键金额字段

### 11.3 风险：停掉 `.log` 文件后排查信息不足

控制方式：

- 保留完整控制台摘要
- 保留 `error_log.csv`
- 保留阶段耗时日志

## 12. 验收标准

本次以同一批真实样本验证：

### 12.1 正确性验收

- `pytest` 全量通过
- `gb`、`sk` 都能成功导出
- workbook 仍包含 8 张 sheet
- `error_log.csv` 仍正常生成
- 关键行数、关键金额、异常高亮结果不回退

### 12.2 产物验收

- `*_处理后.xlsx` 存在
- `*_处理后_error_log.csv` 存在
- `*_处理后.log` 不再生成
- 控制台仍能看到质量摘要

### 12.3 性能验收

重点看相同样本下的以下指标：

- `sk export` 明显低于当前基线 `56.778s`
- `sk total` 明显低于当前基线 `76.202s`
- `gb` 不应因新路由出现明显回退

本次不承诺固定秒数，但验收必须基于同样样本的前后对比，而不是主观感觉。

## 13. 实施顺序建议

### 第一步

先停掉 `*_处理后.log`，保留控制台摘要和 `error_log.csv`。

### 第二步

给 `SheetModel` 补充热点导出提示，使热点大表能直接走快写路径。

### 第三步

对 `成本明细`、`产品数量统计`、`error_log` 切换到轻量快写路径。

### 第四步

对热点大表数据区做样式轻量化，并校准 contract。

### 第五步

重新实跑 `gb/sk`，对比 `export` 和 `total`。

## 14. 结论

这次优化不是再次重做全链路架构，而是一次 **命中真实瓶颈的导出层优化**。

最终方案是：

- 去掉 `*_处理后.log`
- 保留控制台质量摘要
- 保留完整 Excel
- 通过热点大表快写和大表样式轻量化压缩 `sk` 导出时间

这条路线符合当前证据，也符合用户对交付形式和可观测性的约束。
