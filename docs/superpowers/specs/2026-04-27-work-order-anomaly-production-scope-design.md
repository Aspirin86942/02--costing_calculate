# 工单异常分析按生产类型分池设计

## 1. 背景

当前仓库已经有两张与异常分析相关的输出：

- `按工单按产品异常值分析`：工单级异常检测主表，负责 `log`、`Modified Z-score`、异常等级、异常主要来源等核心逻辑
- `按产品异常值分析`：产品级兼容摘要页，当前 `gb` 已支持按 `全部 / 正常生产 / 返工生产` 三种口径拆段展示

现有问题是：`按工单按产品异常值分析` 仍然把同一产品下的正常生产工单与返工生产工单放在同一个异常池中计算 `Modified Z-score`。这会带来两个直接问题：

- 返工工单天然可能存在更高或更不稳定的单位成本，混池后会把正常生产的统计基准拉偏
- 用户在工单异常页无法直接看出某一条工单属于返工还是正常生产，人工复核成本较高

当前业务上已明确的生产类型判定口径如下：

- `汇报入库-普通生产`、`直接入库-普通生产`：认定为 `正常生产`
- `汇报入库-返工生产`：认定为 `返工生产`
- 其他单据类型：不强行归入上述两类

本次目标不是改写异常阈值，也不是改产品级摘要页，而是仅在工单异常页中补齐“生产类型”维度，并让异常值按 `正常生产` 与 `返工生产` 两个池子分开计算。

## 2. 目标

- 在 `按工单按产品异常值分析` sheet 中新增 `生产类型` 列
- `生产类型` 列放在 `工单行` 后、`基本单位` 前
- 复用现有单据类型映射规则，不再定义第二套返工识别逻辑
- 将工单异常页的 `Modified Z-score` 计算口径，从“按产品”改为“按产品 + 生产类型”
- 保留现有加权中位数、加权 MAD、异常阈值和条件格式高亮规则
- 对无法归类到 `正常生产` / `返工生产` 的工单，保留展示，但不参与异常池计算
- 明确把“不可入池”的原因写入 `复核原因`
- 保持产品级 `按产品异常值分析` 的 `全部 / 正常生产 / 返工生产` 逻辑不回退、不改口径

## 3. 非目标

- 不新增新的 workbook sheet
- 不改 `按产品异常值分析` 的布局、分段顺序或聚合口径
- 不改 `gb` / `sk` 当前独立成本项的展示和勾稽规则
- 不调整 `|score| <= 2.5 / <= 3.5 / > 3.5` 的异常分级阈值
- 不把未知单据类型自动合并进 `正常生产` 或 `返工生产`

## 4. 方案对比

### 方案 A：只新增 `生产类型` 展示列，不拆异常池

优点：

- 改动最小
- workbook 契约变更范围最小

缺点：

- 无法解决“返工与正常生产混池”这个核心业务问题
- 统计口径仍然失真，用户只能看到标签，不能得到分池后的异常判断

### 方案 B：新增 `生产类型` 列，并按 `产品 + 生产类型` 分池计算异常值

优点：

- 直接解决混池问题
- 用户可在同一张工单异常页中同时看到生产类型和分池后的异常结果
- 改动范围集中在分析层与契约测试层，不需要重构 ETL 主流程

缺点：

- 分池后样本变少，部分产品在某个池子中可能因有效样本不足而不再产出 `Modified Z-score`
- 工单异常页的列契约、baseline 和相关测试需要同步更新

### 方案 C：把工单异常页拆成 `全部 / 正常生产 / 返工生产` 三个分段或多张表

优点：

- 阅读形式最直观

缺点：

- 改动过大，会扩散到 writer、布局、冻结窗格、条件格式和契约基线
- 超出当前需求，只为区分分池而付出过高实现成本

## 5. 选择

采用方案 B。

原因如下：

- 用户核心诉求是“工单异常页能看出返工/正常生产，并且异常值要拆成两个池子分别计算”
- 方案 B 在不改变 sheet 总体结构的前提下，已经完整满足这个诉求
- 相比方案 C，方案 B 可以把变更控制在分析层和导出契约层，风险更可控

## 6. 设计细节

### 6.1 生产类型映射规则

工单异常页不单独维护映射常量，统一复用当前产品级摘要页使用的单据类型映射规则：

- `汇报入库-普通生产` -> `正常生产`
- `直接入库-普通生产` -> `正常生产`
- `汇报入库-返工生产` -> `返工生产`

其余情况统一映射为 `未归类`，包括：

- `普通委外订单`
- `返工委外订单`
- 未知类型
- 空字符串、空值、脏值

这样做的原因是：

- 避免产品页和工单页出现两套不一致的分类逻辑
- 不把本来没有确认口径的单据类型硬塞入异常池，避免制造伪精确

### 6.2 工单异常页输出列调整

`按工单按产品异常值分析` 的输出列顺序中，新增 `生产类型`：

- 位置：`工单行` 后、`基本单位` 前
- 类型：`text`

不新增原始 `单据类型` 展示列，原因是：

- 当前用户关注的是“返工/正常生产”的业务分类结果，而不是全部原始单据类型值
- 保持 sheet 宽度增长最小，避免无必要地扩大横向滚动范围

### 6.3 异常池分组规则

当前工单异常页对每个指标的 `log` 与 `Modified Z-score` 计算，是按以下维度分组：

- `产品编码`
- `产品名称`

变更后，分组维度调整为：

- `产品编码`
- `产品名称`
- `生产类型`

也就是说：

- `正常生产` 工单只与同产品的 `正常生产` 工单比较
- `返工生产` 工单只与同产品的 `返工生产` 工单比较
- `未归类` 工单不进入任何异常池

### 6.4 `是否可参与分析` 新口径

当前 `是否可参与分析` 实际只反映基础数值条件，例如完工数量和总单位成本是否有效。

本次改动后，该字段调整为更严格的业务含义：

- 完工数量有效
- 总单位完工成本大于 `0`
- `生产类型` 属于 `正常生产` 或 `返工生产`

满足以上三项时，`是否可参与分析 = 是`；否则为 `否`。

这意味着：

- `未归类` 工单即使数量和单位成本都有效，也应标记为 `否`
- `分析覆盖率` 指标会随之下降，但这个下降是口径修正后的真实结果，不是回归

### 6.5 `复核原因` 与空白值规则

对于 `生产类型 = 未归类` 的工单：

- 保留在工单异常页中
- `log_*` 列保持空白
- `Modified Z-score_*` 列保持空白
- 各异常标记列保持空白
- `异常等级` 保持空白
- `异常主要来源` 保持空白
- `复核原因` 追加：`单据类型未归类，不参与正常生产/返工生产异常池`

对于 `正常生产` 或 `返工生产` 工单：

- 仍然先计算 `log`
- 仅当“同产品、同生产类型、同指标”的有效样本数不少于 `3` 条时，才继续计算 `Modified Z-score`
- 若样本数不足 `3` 条，只保留 `log`，其余分数与标记保持空白

这样做的原因是：

- `log` 反映的是单条记录的变换值，只要该条记录自身有效，就有保留价值
- `Modified Z-score` 是群体统计结果，样本不足时强行计算只会制造不可靠分数

### 6.6 异常等级与异常来源汇总规则

`异常等级` 与 `异常主要来源` 继续沿用现有规则：

- 各指标先分别产出 `正常 / 关注 / 高度可疑`
- 再按严重级别和绝对分数汇总出整行的 `异常等级`
- `异常主要来源` 继续选取最严重、且绝对分值最大的来源

补充约束：

- `未归类` 工单由于不进入异常池，因此其所有指标异常标记为空，最终 `异常等级` 与 `异常主要来源` 也必须为空

### 6.7 质量指标影响

对 [src/analytics/quality.py](D:/Program_python/02--costing_calculate/src/analytics/quality.py) 的影响仅体现在现有指标数值变化，不新增指标：

- `工单异常分析输出行数` 口径不变
  - 仍然表示“保留在工单异常页中的工单行数”
- `可参与分析占比` 数值可能下降
  - 因为 `未归类` 工单现在明确不再算作“可参与分析”

这是预期变化，不视为缺陷。

## 9. 输入 / 输出 / 伪代码草案

### 9.1 目标

说明工单异常页如何新增 `生产类型`，并按 `产品 + 生产类型` 分池执行异常检测，同时保证未知单据类型可审计但不误入统计池。

### 9.2 输入

- `work_order_df`
  - 工单级事实数据，至少包含 `product_code`、`product_name`、`order_no`、`order_line`、`completed_qty`、`total_unit_cost`、各指标单位成本列、内部 `doc_type`
- `standalone_metas`
  - 独立成本项元数据，用于保持现有工单异常页列布局兼容
- 现有异常指标配置 `ANOMALY_METRICS`
- 现有单据类型映射规则

### 9.3 输出

- 成功时输出包含 `生产类型`、`是否可参与分析`、`log_*`、`Modified Z-score_*`、异常标记、`异常等级`、`异常主要来源`、`复核原因` 的工单异常页 `FlatSheet`
- 对于 `未归类` 或样本不足的情况，不抛错，而是输出可审计空白结果和对应 `复核原因`
- 不额外返回新的错误对象，不改变现有 workbook writer 的调用方式

### 9.4 伪代码草案

```python
# 目标：
# - 在工单异常页增加“生产类型”
# - 异常值按“产品 + 生产类型”拆成两个池子计算
# - 未归类工单保留展示但不参与异常池

def build_work_order_anomaly_sheet(work_order_df, standalone_metas):
    # 1. 先根据单据类型映射出生产类型，确保产品页和工单页使用同一套业务口径
    work_order_df['production_scope'] = map_doc_type_to_scope(work_order_df['doc_type'])
    work_order_df['生产类型'] = work_order_df['production_scope'].fillna('未归类')

    # 2. 基础可分析条件仍然保留数量和总单位成本有效性校验
    base_can_analyze = (
        work_order_df['completed_qty'] > 0
        and work_order_df['total_unit_cost'] > 0
    )

    # 3. 只有正常生产和返工生产允许入池；未归类必须保留但不能混入任何统计池
    scoped_can_analyze = base_can_analyze and work_order_df['生产类型'].isin(['正常生产', '返工生产'])

    for metric in ANOMALY_METRICS:
        init_log_column(metric)
        init_score_column(metric)

        # 4. 按“产品 + 生产类型”分池，避免返工与正常生产相互污染统计基准
        for group in groupby(work_order_df, ['product_code', 'product_name', '生产类型']):
            if group['生产类型'] == '未归类':
                append_reason(group, '单据类型未归类，不参与正常生产/返工生产异常池')
                continue

            # 5. 只对该指标单位成本 > 0 且数量 > 0 的行计算 log
            valid_rows = filter_positive_metric_and_qty(group, metric)
            fill_log(valid_rows, metric)

            # 6. 样本少于 3 条时停止在 log，避免制造不稳定的群体统计结果
            if len(valid_rows) < 3:
                continue

            median = weighted_median(log_values(valid_rows), qty_weights(valid_rows))
            mad = weighted_mad(log_values(valid_rows), qty_weights(valid_rows), median)

            # 7. MAD 为 0 或缺失时也不继续算分，避免除零或伪异常
            if mad is None or mad == 0:
                continue

            scores = calculate_modified_z_score(valid_rows, median, mad)
            fill_scores(valid_rows, metric, scores)
            fill_flags_from_scores(valid_rows, metric, scores)

    # 8. 汇总是否可参与分析、异常等级、异常主要来源和复核原因
    work_order_df['是否可参与分析'] = scoped_can_analyze
    work_order_df['异常等级'] = summarize_overall_level(work_order_df)
    work_order_df['异常主要来源'] = summarize_primary_source(work_order_df)

    return build_flat_sheet(work_order_df, standalone_metas)
```

### 9.5 风险点 / 边界条件

- 分池后样本数减少，更多记录可能只产出 `log` 而不产出 `Modified Z-score`
- 若未来新增单据类型且业务未及时补充映射，会进入 `未归类`，导致 `可参与分析占比` 下降
- 若产品页和工单页未共用同一映射 helper，后续维护中容易再次出现口径漂移
