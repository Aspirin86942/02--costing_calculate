# 成本工作簿月份区间筛选设计

## 1. 背景

当前仓库的统一入口只有：

- `python main.py gb`
- `python main.py sk`

现状下，管线会按文件名模式挑选一个输入 workbook，然后直接执行完整 ETL，输出：

- `成本明细`
- `产品数量统计`
- 三张 `价量比`
- `按工单按产品异常值分析`
- `按产品异常值分析`
- `*_处理后_error_log.csv`

当前没有“按月份区间筛选工作簿内数据”的运行时能力。

这带来两个问题：

1. 用户无法只针对某几个月份做一次局部核算，只能整本跑完再人工筛。
2. 如果把月份筛选只做在展示层，会导致 `成本明细`、`产品数量统计`、异常分析、质量指标、`error_log` 之间口径不一致。

本次需求已经确认：

- 筛选对象是**工作簿里的数据月份**，不是输入文件名。
- 筛选必须作用于**全链路**，而不是只影响某一张 sheet。
- 多月筛选要支持：
  - 只传开始月
  - 只传结束月
  - 同时传开始月和结束月
- 命令行月份格式只接受 `YYYY-MM`。
- 不传月份区间时，继续处理全部月份，保持当前行为。
- 如果月份区间合法但工作簿里没有任何命中数据，仍然输出空结果，不报失败。
- 传了月份参数后，输出文件名要自动追加月份后缀，避免覆盖默认结果。

## 2. 目标

- 在统一入口增加可选的月份区间参数：
  - `--month-start YYYY-MM`
  - `--month-end YYYY-MM`
- 将月份筛选固定为**工作簿数据级别**的全链路过滤能力。
- 保证过滤后的口径在以下产物中完全一致：
  - `成本明细`
  - `产品数量统计`
  - 三张 `价量比`
  - `按工单按产品异常值分析`
  - `按产品异常值分析`
  - 质量指标
  - `error_log.csv`
- 保证不传月份参数时，现有行为和现有输出不发生变化。
- 保证空命中时仍能产出结构完整的 workbook 和 `error_log`。

非目标：

- 不改现有异常分析算法本身。
- 不改变 `gb` / `sk` 的既有业务规则。
- 不引入新的交互方式，例如 GUI、配置文件或 Excel 内嵌筛选按钮。

## 3. 方案对比

### 方案 A：CLI 传参后，在标准化成本表阶段做行级月份过滤

做法：

- `main.py` 解析月份区间参数
- 参数先归一化为规范月份值
- workbook 读入后，先完成标准化，再在 `NormalizedCostFrame` 上按月份过滤
- 过滤后的结果继续进入拆表、分析、写出链路

优点：

- 口径最一致，所有 downstream 产物天然共享同一份过滤结果
- 过滤点单一，后续维护时不容易漏掉某个分析分支
- 与当前 Polars 主链路贴合，适合做日志和质量指标补充

缺点：

- 需要新增一个运行时配置对象，把月份区间从 CLI 一直传到 pipeline
- 需要补充空结果保护测试

### 方案 B：完成拆表后分别过滤 `detail_df` / `qty_df`

做法：

- workbook 标准化和拆表照旧执行
- 在 `split_result` 形成后分别过滤 `detail_df` 和 `qty_df`
- 后续分析继续使用过滤后的两张表

优点：

- 看起来改动点少

缺点：

- 过滤逻辑分散
- 如果后续新增直接依赖 `NormalizedCostFrame` 的分支，容易漏过滤
- 对长期维护不利

### 方案 C：只在分析页或写 Excel 前过滤

做法：

- 前面全量跑完
- 最终只在分析页或 writer 层筛月份

优点：

- 表面改动最小

缺点：

- 与“全链路同口径”目标直接冲突
- `成本明细`、质量指标、`error_log` 与分析页会不一致
- 业务上不可接受

## 4. 选择

采用方案 A。

本次能力本质上是“运行时月份区间过滤”，不是“展示层筛选”。因此过滤必须尽量靠前，但又必须发生在月份已经标准化、可比较之后。

也就是说：

- 不在文件名匹配阶段做月份筛选
- 不在拆表之后重复散落过滤
- 不在 Excel 写出阶段临时裁剪
- 只在标准化成本表阶段做一次正式过滤

## 5. 设计细节

### 5.1 参数与业务语义

CLI 新增两个可选参数：

- `--month-start YYYY-MM`
- `--month-end YYYY-MM`

语义固定如下：

- 两个都不传：处理全部月份
- 只传 `month_start`：保留 `period >= month_start`
- 只传 `month_end`：保留 `period <= month_end`
- 两个都传：保留 `month_start <= period <= month_end`

这里明确采用**双闭区间**语义，而不是闭开区间。

原因是：

- 本次筛选对象是离散月份，不是连续时间戳
- 业务侧直接理解“从 `2025-01` 到 `2025-03` 包含 1 到 3 月”更直观
- 不需要在用户输入和内部实现之间做“结束月 +1 月”的额外换算

参数格式只接受 `YYYY-MM`。

以下输入应直接判为非法：

- `2025-1`
- `2025-13`
- `2025-00`
- `2025/01`
- 任意非 `YYYY-MM` 形式的文本

如果同时传入两个边界，还必须满足：

- `month_start <= month_end`

不满足时应在进入 ETL 前直接报错退出。

### 5.2 运行时配置对象

不建议在 `main.py`、`runner.py`、`costing_etl.py`、`pipeline.py` 间散落传递两个裸字符串。

建议新增一个轻量运行时对象，例如：

- `MonthRange`

建议职责：

- 保存 `start: str | None`
- 保存 `end: str | None`
- 保证实例化后内部值已经是合法的规范月份 `YYYY-MM`
- 提供区间合法性检查
- 提供命名后缀生成能力
- 提供便于日志输出的文本描述

建议新增位置：

- `src/etl/month_filter.py`

这样做的原因是：

- 这是运行时 ETL 能力，不是静态管线配置
- 不需要污染 `PipelineConfig`
- CLI、runner、ETL、输出命名都可以共用同一个对象

### 5.3 规范月份键与复用规则

过滤比较不能直接基于展示列 `月份=2025年01期` 做字符串判断。

必须统一基于规范月份键：

- `period_key = YYYY-MM`

仓库里已经存在月份标准化逻辑：

- `src.analytics.fact_builder.normalize_period`

本次实现应复用同一套标准化规则，而不是在 CLI 解析、ETL 过滤、分析阶段各写一份正则。

设计约束是：

- 用户输入月份区间时，先标准化成 `YYYY-MM`
- workbook 中每一行要用于过滤的月份，也先标准化成 `YYYY-MM`
- 最终比较永远只发生在 `YYYY-MM` 上

这样可以保证：

- CLI 参数和工作簿里的 `年期` / `月份` 展示格式不会互相干扰
- 边界月比较不会因为中文展示格式而出错

### 5.4 过滤落点与数据流

建议数据流固定为：

1. `main.py` 解析月份参数
2. 构造 `MonthRange | None`
3. `run_pipeline(config, month_range)` 把运行时筛选配置继续往下传
4. `CostingWorkbookETL` 持有 `month_range`
5. 在 `CostingEtlPipeline.build_workbook_payload(...)` 中：
   - 先读取 raw workbook
   - 再执行 `build_normalized_cost_frame(...)`
   - 再对 `NormalizedCostFrame` 做月份过滤
6. 过滤后的 `NormalizedCostFrame` 才进入：
   - `split_normalized_frames(...)`
   - `build_report_artifacts(...)`
   - `build_sheet_models(...)`

过滤应发生在：

- 标准化完成之后
- 拆表之前

不建议发生在：

- 文件发现阶段
- `detail_df` / `qty_df` 形成之后
- writer 层

原因是：

- 此时列名、汇总行清理、向下填充都已经完成，月份字段可稳定读取
- 此时 downstream 尚未分叉，单点过滤即可统一全部产物口径
- 改动边界最清晰

### 5.5 过滤算法约束

在过滤阶段，可以临时生成一个辅助列，例如：

- `period_key`

比较规则固定为：

- 只传 `start`：`period_key >= start`
- 只传 `end`：`period_key <= end`
- 双边都传：`start <= period_key <= end`

过滤完成后：

- `period_key` 不应暴露到最终 Excel 列结构
- 它只作为中间过滤辅助键使用

这里要特别避免一类错误：

- 一部分逻辑拿 `月份` 展示列做比较
- 另一部分逻辑拿分析阶段的 `period` 做比较

如果这样混用，会导致边界月命中行为不一致。

本次设计要求：

- 过滤只做一次
- 过滤只认规范月份键

### 5.6 输出命名规则

不传月份参数时：

- workbook 继续使用现有命名：`xxx_处理后.xlsx`
- `error_log.csv` 继续使用现有命名：`xxx_处理后_error_log.csv`

传了月份参数时，输出文件名必须自动追加月份后缀，避免覆盖默认结果。

建议规则：

- 只传 `--month-start 2025-01`
  - workbook：`xxx_处理后_from_2025-01.xlsx`
  - error log：`xxx_处理后_from_2025-01_error_log.csv`
- 只传 `--month-end 2025-03`
  - workbook：`xxx_处理后_to_2025-03.xlsx`
  - error log：`xxx_处理后_to_2025-03_error_log.csv`
- 同时传 `--month-start 2025-01 --month-end 2025-03`
  - workbook：`xxx_处理后_2025-01_2025-03.xlsx`
  - error log：`xxx_处理后_2025-01_2025-03_error_log.csv`

`MonthRange` 建议直接提供后缀生成方法，避免命名规则散落在 `runner.py`。

### 5.7 错误处理与空结果策略

错误处理分为两类：

#### 5.7.1 参数错误

包括但不限于：

- 非法月份格式
- 非法月份值
- `month_start > month_end`

这些错误属于**输入契约错误**，应在进入 ETL 前直接报错退出。

行为要求：

- 不读取 workbook
- 不生成输出文件
- 返回失败状态码
- 控制台输出清晰错误信息

#### 5.7.2 数据结果为空

指的是：

- 参数合法
- 输入文件存在
- 但工作簿中没有任何一行命中月份区间

这类情况不是失败，而是**合法的空业务结果**。

行为要求：

- 继续生成 workbook
- 继续生成 `error_log.csv`
- workbook 保留既有 sheet 结构，但数据区为空
- `error_log.csv` 保持稳定表头结构
- `run_pipeline(...)` 返回成功状态码 `0`
- 控制台和日志明确打印“月份过滤后无命中数据”

### 5.8 质量指标与可观测性

月份过滤上线后，控制台摘要至少要额外体现以下信息：

- 本次月份区间参数
- 过滤前总行数
- 过滤后总行数
- 过滤前涉及月份集合
- 过滤后涉及月份集合

当结果为空时，质量指标不能静默失败。

建议原则：

- 行数勾稽继续输出，明确显示过滤前后变化
- 空值率、唯一性、分析覆盖率等指标若分母为 0，值应写为可审计表达，例如：
  - `N/A`
  - `0/0`
- `description` 里要明确写出“月份过滤后无数据，指标不适用”

目标不是让空结果“看起来像成功”，而是让空结果**可解释、可审计**。

### 5.9 对下游空数据契约的要求

为了保证“空命中仍输出成功”，以下链路都必须支持 0 行输入：

- `split_normalized_frames(...)`
- `build_report_artifacts(...)`
- `build_sheet_models(...)`
- workbook writer
- `error_log` 导出

这些路径必须在 0 行时返回**结构完整对象**，而不是因为：

- 缺列
- groupby 结果为空
- pivot 空表
- 排序空表
- 条件格式范围假设有数据

而中途报错。

本次设计把“空数据仍可写出稳定结构”视为正式契约，不是额外兼容。

### 5.10 测试设计

至少补以下测试组：

#### 5.10.1 参数解析测试

- 两个参数都不传时，`month_range is None`
- 只传 `month_start`，能正确解析
- 只传 `month_end`，能正确解析
- 同时传 `month_start` / `month_end`，能正确解析
- 非法格式报错
- 非法月份值报错
- `month_start > month_end` 报错

#### 5.10.2 标准化过滤单测

给一个包含多个不同月份的 `NormalizedCostFrame`，验证：

- 只传开始月时的保留范围
- 只传结束月时的保留范围
- 双边区间时的保留范围
- 边界月份命中行为符合双闭区间

#### 5.10.3 全链路口径测试

输入同时含：

- `2025-01`
- `2025-02`
- `2025-03`

指定 `2025-02` 到 `2025-03` 后，验证：

- `成本明细` 只剩 2 月和 3 月
- `产品数量统计` 只剩 2 月和 3 月
- 异常分析页只基于过滤后的数据
- 质量指标的行数勾稽反映过滤后的口径

#### 5.10.4 空命中测试

例如筛选：

- `2026-01` 到 `2026-03`

验证：

- 处理成功
- workbook 文件生成成功
- `error_log.csv` 生成成功
- 所有目标 sheet 仍存在
- 数据区为空但表头结构稳定

#### 5.10.5 输出命名测试

验证：

- 默认无月份参数时，命名保持现状
- 只传开始月时，命名追加 `_from_YYYY-MM`
- 只传结束月时，命名追加 `_to_YYYY-MM`
- 双边区间时，命名追加 `_YYYY-MM_YYYY-MM`

#### 5.10.6 回归测试

验证：

- 不传月份参数时，现有行为完全不变
- 现有无月份相关测试不需要因为新功能而整体改写其业务预期

## 6. 影响范围

按当前设计，主要影响文件预计为：

- `main.py`
- `src/etl/runner.py`
- `src/etl/costing_etl.py`
- `src/etl/pipeline.py`
- `src/etl/month_filter.py`（新增）
- 与月份过滤相关的测试文件

如果实现阶段发现 `split_normalized_frames(...)` 或后续分析链路对空数据支持不足，还需要补空结果保护，但不应改变本设计的核心边界。

## 7. 风险与控制

主要风险有两个：

1. 现有某些聚合或透视逻辑默认“至少一行数据”，空命中时可能暴露出隐藏问题。
2. 仓库同时存在展示月份和内部月份概念，如果过滤不统一走规范月份键，边界月可能出现不一致。

控制方式：

- 实现时优先补空数据路径测试
- 明确规定过滤只基于 `YYYY-MM` 规范月份键
- 不允许在 writer 或展示层再做第二次月份判断

## 8. 验收标准

满足以下条件即可视为本设计落地完成：

- CLI 可接受可选的 `--month-start` / `--month-end`
- 参数语义符合本 spec
- 过滤发生在标准化成本表阶段
- 所有下游产物与质量指标共享同一份过滤口径
- 空命中时成功输出空 workbook 和空 `error_log`
- 传月份参数时输出文件自动带月份后缀
- 不传月份参数时行为与现状一致
